# LightGBM + SHAP 실무 아키텍처 가이드

------------------------------------------------------------------------

# 1. 전체 시스템 아키텍처

    [DART / 시장 데이터 수집]
            ↓
    [Raw Data 저장 (DB)]
            ↓
    [Feature 생성 파이프라인]
            ↓
    [DEAL_FEATURE 테이블]
            ↓
    [ML 학습 (LightGBM)]
            ↓
    [모델 저장]
            ↓
    [실시간 예측 API]
            ↓
    [SHAP 설명 생성]
            ↓
    [결과 제공 / 리포트]

------------------------------------------------------------------------

# 2. 구성 요소 상세

## 2.1 데이터 수집

-   공시 데이터 (DART)
-   주가 데이터
-   재무제표

------------------------------------------------------------------------

## 2.2 Feature 파이프라인

    Raw Data → 가공 → Feature 생성 → DB 저장

예: - 할인율 - 부채비율 - 시총 대비 발행 규모

------------------------------------------------------------------------

## 2.3 Feature 저장 구조

    DEAL_FEATURE
    - deal_id
    - discount_rate
    - debt_ratio
    - size_ratio
    - market_return
    - label

------------------------------------------------------------------------

# 3. ML 학습 구조

    DB → Pandas → LightGBM → 모델 파일

-   학습 주기: 하루 1회 or 주기적
-   모델 파일: deal_model.pkl

------------------------------------------------------------------------

# 4. 실시간 예측 구조

    Spring Boot
       ↓
    REST API 호출
       ↓
    Python ML Server
       ↓
    LightGBM
       ↓
    SHAP
       ↓
    응답 반환

------------------------------------------------------------------------

# 5. API 흐름

## 요청

    POST /predict

    {
      "discount_rate": 0.15,
      "debt_ratio": 0.8,
      "size_ratio": 0.3
    }

------------------------------------------------------------------------

## 응답

    {
      "success_prob": 0.78,
      "label": "SUCCESS",
      "top_factors": [
        {"feature": "discount_rate", "impact": 0.25},
        {"feature": "debt_ratio", "impact": -0.18}
      ]
    }

------------------------------------------------------------------------

# 6. SHAP 리포트 흐름

    예측 결과 → SHAP 계산 → 영향도 정렬 → 텍스트 변환

예: "할인율이 높아 성공 확률 상승"

------------------------------------------------------------------------

# 7. 배치 vs 실시간

## 배치

-   모델 학습
-   Feature 재계산

## 실시간

-   딜 발생 시 예측

------------------------------------------------------------------------

# 8. Kafka 기반 확장

    deal-event → feature 생성 → predict → 결과 저장

------------------------------------------------------------------------

# 9. 운영 구성 (Docker)

    - spring-api
    - ml-server (python)
    - database
    - kafka (optional)

------------------------------------------------------------------------

# 10. 장애 대응

-   ML 서버 다운 → fallback
-   예측 실패 → 기본값 반환

------------------------------------------------------------------------

# 11. 성능 최적화

-   Feature 캐싱
-   인덱스 최적화
-   모델 메모리 상주

------------------------------------------------------------------------

# 12. 핵심 설계 원칙

1.  ML 서버 분리
2.  Feature 계약 고정
3.  데이터 누수 방지
4.  설명 가능성 확보

------------------------------------------------------------------------

# 13. 최종 요약

    데이터 → Feature → 모델 → 예측 → 설명
